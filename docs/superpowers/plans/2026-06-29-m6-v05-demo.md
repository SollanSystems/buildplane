## Pre-build resolutions (close before dispatching slices)

1. **`riskClass` rubric (fixes "arbitrary thresholds" risk).** Computed in `validate.ts` from existing signals:
   `unsafeReasons.length > 0` or `status=UNSAFE_TO_RUN` ‚Üí **`high`**; else `missingEvidence.length > 0` or `status=INSUFFICIENT_EVIDENCE` ‚Üí **`medium`**; else clean `PASS` **with** non-empty `allowedSideEffects`/writable globs ‚Üí **`medium`**; else clean `PASS` no side-effects ‚Üí **`low`**. Pinned by a test table before coding.
2. **`result_ready` is built by the dogfood run, reviewed by ceremony (M6-S6).** The planning session does **not** hand-build the kind; it stages the run. Fallback (spec ¬ß7): bounded-retry then ceremony-build. The **emit** wiring (M6-S7) is hand-built **after** the dogfood `result_ready` merges.
3. **Inbox feed stays derived; `result_ready` coexists** (spec Decision 1) ‚Äî no inbox-query rewrite in M6.
4. **Net-egress is declarative-only by default** (spec Decision 3) ‚Äî enforcement sub-slice is spike-gated.
5. **Property 2 is demonstrated against a command packet, not the Claude worker** (spec ¬ß4) ‚Äî no Claude-subprocess hard-enforcement on the critical path.
6. **Two live runs are operator-triggered** (spec ¬ß7) ‚Äî slices stage, the operator fires.
7. **`ResultReadyPort` design** (spec ¬ß5 resolution 1a) ‚Äî a **new `ResultReadyPort`** (`recordResultReady(runId, admissionEventId, acceptanceEventId): Promise<void>`) injected into the orchestrator alongside `acceptancePort`, NOT an extension of `BuildplaneAcceptancePort.recordAcceptance`. Concrete emitter mirrors `ledger-operator-decision.ts`; wired in `apps/cli` at the `acceptancePort` injection site. Locked before S7 is dispatched.

   > **Amended 2026-07-01 (A1):** the `recordResultReady` emit is gated on the run's **terminal outcome** ‚Äî it fires only **after** the terminal advance/passed decision from `policy.evaluateRun`, **not** when `acceptancePort.recordAcceptance` resolves `outcome='passed'` (the per-attempt ~orchestrator.ts:799 point). The 2026-07-01 live dogfood run recorded an `acceptance_recorded(passed)` + `plan_receipt(failed)` pair on one tape ‚Äî a per-attempt acceptance pass can still terminate `failed`, so emitting at :799 would sign a false `result_ready` onto the L0 tape. See M6-S7 for the moved injection point.
8. **`run_completed` field source ‚Äî synchronous, option (a)** (spec ¬ß5 resolution 1b) ‚Äî `RunCompletedV1.{duration_ms,event_count,unit_count}` are supplied **synchronously** from the already-loaded `snapshot = storage.inspectTarget(runId)` (`duration_ms = Date.now() - run.createdAt`, `event_count` from a sync SQL `COUNT`, `unit_count` from the snapshot). `applyOperatorDecisionSideEffect` stays **synchronous** (`void`) ‚Äî its callers at `orchestrator.ts:2166` and `2178` are NOT promoted to `await`, preserving the D2‚ÄìD4 crash-recovery marker ordering. The async-storage-query alternative (option b) is explicitly rejected. Locked before S7 is dispatched.

   > **Amended 2026-07-01 (A3 ‚Äî U64 wire serialization):** separate from the sync field-*source* "option (a)" locked above, the U64‚ÜíTS `number` precision hazard is resolved with **per-field string serialization for `RunCompletedV1` only**: `duration_ms`/`event_count`/`unit_count` serialize as **`String`** on the signed wire, so Rust‚ÜîTS digests are byte-identical and the RED "U64 guard" holds. The global `typeshare.toml` `U64 = "number"` mapping is **untouched** (`checkpoint.rs::TapeCheckpointV1`, `model_io.rs::ModelResponseV1` keep `number`). Safe with **no tape migration**: a repo-wide scan (2026-07-01) of the only real ledger DB (`.worktrees/m6-s6-result-ready-dogfood/.buildplane/ledger/events.db`) found **`run_completed` count = 0** ‚Äî the struct is in the wire map but was never emitted/signed. Option (b) (keep `number` + a 2^53-range rationale) is rejected.
