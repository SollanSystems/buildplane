## Design

- `runPlanForgeDispatchCommand` never passed `onClaudeToolEvent`, so the plumbing that already threads `opts.onClaudeToolEvent` → executor `onToolEvent` (loadCliOrchestrator ~1448/~1787) was dormant on dispatch.
- Dispatch brackets each packet with `activity_started`/`activity_completed` (kernel `LedgerActivityPort`), NOT `unit_started`/`unit_completed` (the `bp run` path). The tool sink needs the `activity_started` EVENT id as the tool_request parent, but that id is auto-assigned inside the emitter and invisible to a plain port wrapper.
- `createDispatchToolUnitTracker` observes the `TapeEmitter` feeding the activity port: it assigns + records an explicit id on each `activity_started`, and clears the active ctx on `activity_completed`. The dispatch loop supplies each packet's `unit.id` via `beginUnit` before `runPacketAsync` (packets dispatch sequentially, so one mutable ctx is safe).
- The tool sink (`createClaudeToolLedgerEmitter`) is bound to the RAW `emitter` (same serialized signed writer as activity/acceptance/result_ready events); only the activity port sees the observed proxy, which delegates every other emit + lifecycle call unchanged. Composes with `withCrashAfterActivityGuard` transparently (guard wraps the port, tracker wraps the emitter the port uses) — guard behavior unchanged.
- `projectRoot` (`working_directory`) = the dispatch `workspace` (== `resolve(cwd)`), matching the `bp run` precedent (`ledgerWorkspacePath`).
